# Рекомендации по подготовке Markdown для RAG

Чтобы файлы хорошо chunk-ились и давали релевантные результаты, соблюдай эти правила.

## ✅ Заголовки

- Начинай каждый смысловой блок с заголовка `##` или `###`
- Не пропускай уровни (`#` → `###` без `##`)
- Заголовки должны быть осмысленными — по ним строится контекст

```md
## Раздел темы
### Подраздел
Текст подраздела ...
```

## ✅ Размер секций

Одна секция = один чанк (до 1200 символов). Если секция длиннее — она будет принудительно разбита.

- Оптимально: 800–1200 символов
- Минимум: 800 (секции короче 800 символов автоматически склеиваются с соседними)
- Максимум: 1200 символов (настраивается в `rag-sources.yaml`)

## ✅ Списки

- Используй маркированные (`-`) или нумерованные (`1.`) списки
- Избегай вложенных списков глубже 2 уровней

## ✅ Таблицы

Допустимы, но не делай таблицы шире 5 столбцов и длиннее 20 строк — они могут не вместиться в один чанк.

## ✅ Код

```md
Блоки кода обрамляй тройными обратными кавычками с указанием языка:
\`\`\`python
print("hello")
\`\`\`
```

## ✅ Разделяйте технологии по файлам

Чтобы RAG не путал языки и технологии — раскладывайте их по разным `.md` файлам.
В одном файле — одна технология / одна тема.

```
❌ Плохо: один огромный tech-rules.md со всем подряд
✅ Хорошо: java-rules.md, python-rules.md, kotlin-rules.md — каждый о своём
```

Когда все правила в одном файле, поиск находит весь файл целиком и LLM путает технологии.
Разделение по файлам повышает точность: запрос "java final" → найдёт `java-rules.md`, а не общий файл.

## ❌ Чего избегать

- HTML-разметки (будет воспринята как текст)
- Спецсимволов вроде `\{` без экранирования
- Секций без заголовков (упадут в дефолтный чанк)
- Файлов длиннее 100 КБ (лучше разбить на несколько)

## ✅ Стоп-слова

При поиске из запроса автоматически удаляются стоп-слова — частотные слова без смысловой нагрузки.
Список используемых стоп-слов:

```
Русские: какой, какая, какие, какое, каких, каким, что, как, где, когда, почему,
         который, которая, которое, которые, быть, есть, иметь, должен, должна
Английские: is, are, the, a, an, of, in, to, for, and, or, with, at, by, from, as, on
```

Это значит, что контент инструкций должен содержать **ключевые термины**, а не общие фразы — иначе чанк не найдётся.

### Пример

```
❌ Длинный запрос со стоп-словами:
   "Какие есть правила по тому как нужно оформлять код на java?"

   → после фильтрации останется: "правила оформлять код java"
   → слова "правила", "оформлять", "код", "java" ищутся в инвертированном индексе
   → все остальные слова (какие, есть, по, тому, как, нужно, на) отброшены как стоп-слова

✅ Короткий запрос (только ключевые слова):
   "правила оформления java код"

   → после фильтрации останется: "правила оформления java код"
   → те же ключевые слова, шум отсутствует изначально
   → выше точность поиска
```

Вывод: формулируй контент так, чтобы в нём были содержательные термины, а запросы к нему — без лишних слов.
